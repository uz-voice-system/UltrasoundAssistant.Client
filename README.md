# UltrasoundAssistant.DoctorClient

Клиентское приложение врача-диагноста на **Avalonia UI + .NET 8 + MVVM** для работы с API шлюзом системы голосового помощника УЗИ.

## Что реализовано

- авторизация врача через `POST /api/auth/login`;
- просмотр списка пациентов;
- просмотр шаблонов исследований;
- просмотр списка отчётов;
- создание черновика отчёта;
- отправка распознанного голосового фрагмента на сервер;
- ручное редактирование полей шаблона;
- сохранение полей и завершение отчёта.

## Куда загружать модель Whisper

Локальная модель подключается на клиенте.

По умолчанию путь указан в `appsettings.json`:

```json
"Speech": {
  "ModelPath": "Assets/Models/ggml-base.bin",
  "Language": "ru"
}
```

### Что сделать

1. Возьми модель в формате, совместимом с `whisper.cpp`.
2. Положи файл модели сюда:

```text
UltrasoundAssistant.DoctorClient/Assets/Models/ggml-base.bin
```

3. Если имя файла другое, поменяй `Speech:ModelPath` в `appsettings.json`.

В `whisper.cpp` модели загружаются отдельно от кода приложения, а сами модели скачиваются как совместимые файлы для запуска локальной транскрипции. citeturn978739search1turn978739search3

## Что нужно заменить у тебя в проекте

Сейчас в `WhisperSpeechService` стоит безопасная заглушка `ProcessTextStubAsync()`, чтобы можно было проверить весь pipeline клиента даже без native-интеграции.

Когда будешь подключать реальный `whisper.cpp`, нужно заменить тело метода в файле:

```text
Services/WhisperSpeechService.cs
```

на вызов твоей .NET-обёртки над `whisper.cpp`.

## Какие серверные эндпоинты использует клиент

- `POST /api/auth/login`
- `GET /api/patients`
- `GET /api/templates`
- `GET /api/reports`
- `POST /api/reports`
- `POST /api/reports/field`
- `POST /api/reports/voice`
- `POST /api/reports/complete`

## Архитектурная привязка к ВКР

Проект соответствует задачам проектного практикума:

1. **Архитектурный проект клиент-серверной системы** — клиент работает с API Gateway, серверная часть построена по схеме CQRS + Event Sourcing, где запись и чтение разделены, а чтение идёт из Projection Service. fileciteturn0file3
2. **База данных для пациентов, шаблонов и отчётов** — клиент использует именно эти сущности, которые уже выделены в модели БД (`patients`, `templates`, `reports`, `keywords`). fileciteturn0file3
3. **Сопоставление ключевых слов с полями шаблона** — по архитектуре это делается через таблицу `keywords`, где фраза врача связывается с целевым полем шаблона. fileciteturn0file3
4. **Формирование, редактирование и сохранение отчёта** — в клиенте есть сценарий: выбрать пациента → выбрать шаблон → создать черновик → отправить голосовой текст → вручную исправить поля → завершить отчёт. Это соответствует use case врача. fileciteturn0file2
5. **Клиентское приложение врача-диагноста** — реализовано отдельным desktop-клиентом на Avalonia, как и указано в задании на проектный практикум. fileciteturn0file0
